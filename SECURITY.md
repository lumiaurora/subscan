# Security Policy

## Supported versions

Security fixes are best-effort for:

- the latest tagged release
- the current `main` branch

Older releases may not receive fixes.

## Reporting a vulnerability

Please do not open public GitHub issues for security problems.

Use one of these private paths instead:

- GitHub private vulnerability reporting or a draft security advisory, if available
- direct contact with the maintainer through GitHub

Please include:

- a clear description of the issue
- affected version or commit
- reproduction steps or proof of concept
- impact assessment
- any suggested fix or mitigation if you have one

## Response expectations

Best effort targets:

- initial acknowledgement within 72 hours
- triage and severity assessment within 7 days
- a fix or mitigation timeline communicated after confirmation

These are targets, not guarantees.

## Scope notes

`subscan` is a passive enumeration tool. Reports that require adding active exploitation or out-of-scope invasive behavior to demonstrate impact are usually not needed. Clear reproduction details and affected code paths are more useful.

## Disclosure

Please wait for a fix or coordinated disclosure guidance before sharing details publicly.
